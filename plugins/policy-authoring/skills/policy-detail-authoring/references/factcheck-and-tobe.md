# 팩트체크 택소노미 · 체크리스트 · to-be 후보

PI 값의 확실성을 분류하고(가/나/다), 불확실 항목을 붉은 배지로 표면화하고, to-be 개선안을 별도 문서로 아카이빙하는 방법과 출력 템플릿.

## 확실성 택소노미 (가/나/다)

| 축 | 의미 | 조치 |
|---|---|---|
| **가 = 확정** | as-is 명문 + 시스템 무관 business rule. 운영 일정·정책 규칙 등 데이터 변동 없는 값. | 표기 없음(평범하게 서술) |
| **나 = 시스템 이관 미확정** | 값·필드·코드체계가 특정 시스템 산출물 → 시스템 이관(교체) 후 동일 필드/값 유지 미확정 | **붉은 "현업 검토 필요" 배지** + 질문리스트. `field_review`에 사유. |
| **다 = to-be 미정** | as-is 근거 없이 to-be 설계가 정의 중인 값(임계값·갱신주기·산식·매핑 등) | 배지 + 질문리스트. `field_review`에 사유. |

> *예(BIL)*: 나 = "SWING 무선 24코드가 BSS 이관 후 동일한가?" / 다 = "통합 한도 전환 비율 미정".

### 플래깅 경계 (노이즈 방지)
- **플래그한다**: 값/코드/필드가 **시스템 산출물**인 항목(나), **실질적 to-be 미정 값**(다).
- **플래그 안 한다**: 사소한 구현 디테일(전화번호·API 코드·백엔드 트랜잭션), 불확실성 낮은 계산값. 시스템 의존 전수 플래깅은 노이즈 → "값/코드/필드가 시스템 산출물인 항목"으로 한정.
- 시스템 의존 단서는 보통 기존 근거의 "현업 확인/미정/미확정/권고안/초안" 표현에서 출발 → 보이는 배지로 표면화.
- **범위 경계**: field_review는 **그 값을 실제로 관할하는 PI**에 단다. 본문(rule/criteria)에 등장하지 않는 인접 기능의 미정 값은, 그 값을 다루는 PI 쪽에 플래그한다(커버리지 '가짜 매핑 금지'와 같은 정신) — 한 청크에서 타 소관 값을 억지로 끌어오지 말 것.

## 7항목 팩트체크 체크리스트
각 PI를 아래로 1차 스크리닝(키워드 매칭은 ②④ 오탐 많음 → ③⑤ⓐ를 진짜 약신호로 신뢰):

| # | 기준 | 정의 |
|---|---|---|
| ① 조건 | 적용 조건(~면/경우/때/이상/이내)이 명확한가 |
| ② 대상 | 주체(고객/운영자) 또는 객체(계정/수단)가 있는가 |
| ③ 시스템 기준 | 출처/근거 시스템(원장·시스템·연계)이 식별되는가 |
| ④ 화면 동작 | 화면 거동(표시/노출/비활성/안내/제공)이 있는가 |
| ⑤ 예외 시 대체 | 예외 + 복구 경로(재시도/대체/문의)가 명시됐는가 |
| ⓐ 수치 보존 | as-is 수치 패턴(개월/일/%/회/년)이 보존됐는가 |
| ⓑ 어조 | 단순 평서 1문장 + 기준 ≤1/5 충족이면 약한 어조로 플래그 |

등급(참고): A(4+ 충족, ①·⑤ 둘 다) / B(2~3) / C(0~1 또는 약어조).

## to-be 수정후보 (별도 문서 — 본문 직접수정 금지)

현 PI를 to-be로 **개선**할 net-new 후보는 정책 본문을 고치지 않고 **별도 문서에만** 기록(사용자가 추후 직접 반영). "미정 값(현업 확인 대상=field_review)" ≠ "개선 제안(to-be 후보)" — 둘을 구분한다.

### to-be 6대 지향 (근거 축)
① 금액 이해(고객 언어 재구성) ② 사전 알림 ③ 변경 영향 예측 ④ 셀프 완결 원스톱 ⑤ 가족·다회선·위임 ⑥ 24시간 안심.

## 출력 문서 템플릿

### A. 현업 검토 질문리스트 — `audit/field_review_questions_YYYY-MM-DD.md`
spec의 `field_review` 항목을 질문으로 정리. 컬럼:

| 번호 | PI (ID·명) | 축 | 현재 가정 | 현업 확인 질문 | 근거 | 상태 |
|---|---|---|---|---|---|---|
| FR-XXX-01 | -12·청구항목 분류 | 나 | as-is 24코드 | 이관 후 동일 코드체계? | as-is 섹션 2.1.3 | 🟡열림 |

- 축 = 나/다. 상태 = 🟡열림 / 🔵답변대기 / ✅반영.
- PG별 소계(나 N건·다 N건) + 전체 종합.
- **정합 가드**: 문서의 질문 수 == spec의 `field_review` 수.

### B. to-be 수정후보 — `audit/tobe_revision_candidates_YYYY-MM-DD.md`
컬럼:

| 번호 | PI (ID·명) | 현재(as-is) | to-be 수정 제안 | 근거(지향) | 비고 |
|---|---|---|---|---|---|
| TB-XXX-01 | -12·청구항목 분류 | 24코드→5분류 | 5분류에 '쉬운 설명'+예시 부여 | ① 금액 이해 | FR-XXX-04 연계 |

- ⚠️ **본문 미수정** — 후보만 기록. net-new 없으면 "net-new 없음"으로 명시(이미 to-be 반영됐거나, 미정 항목은 field_review로 관리).
