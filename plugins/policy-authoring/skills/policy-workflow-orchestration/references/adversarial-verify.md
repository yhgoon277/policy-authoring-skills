# 적대 검수 패턴 (Adversarial Verification After a Batch)

큰 묶음(BL 보강 다건·PI 배치·커버리지 보강) 완료 후, **편집자와 분리된 독립 에이전트**로 `계획 vs 실제 diff` 충실도를 검증하는 cross-cutting 규율. 자기 작업을 자기가 검수하면 확증편향이 끼므로, **신규 컨텍스트**(이 대화의 기억 없음)로 돌린다.

> **참조 구현**: hub 16 BL 보강 후 독립 에이전트 4개 병렬 → 4/4 PASS. NC G2·G5 해소 후 각각 10/10 PASS.

## 언제
- 한 묶음에서 여러 노드를 건드렸을 때(단건 편집 1건 루프로는 한눈에 안 잡힘).
- 좋은 타이밍: 감사 PASS 직후 / 요약·핸드오프 직전.

## 어떻게
1. **분리**: 편집한 에이전트가 아닌 새 에이전트(들). 독립 과제 ≥2면 병렬로.
2. **입력**: `BACKLOG/플랜` + `실제 git diff`(또는 재빌드 spec) 만 준다. 작성 의도 설명 금지(에이전트가 스스로 대조).
3. **표적**: 과대평가(=충실히 반영했다는 거짓 주장)가 최대 위험 → '충실 반영' 주장을 집중 검증.

## 체크리스트 (전부 PASS 목표)
- [ ] **요구/계획 충실 반영** — BL 각 항목이 실제 diff에 들어갔는가(이름만/미반영 아님).
- [ ] **제안 누락 0** — 계획에 있던 보강이 빠지지 않았는가.
- [ ] **노드 오편집 0** — 대상 외 노드를 건드리지 않았는가(diff 범위 일치).
- [ ] **날조 0** — 없는 값·수치·"공통 N종" 같은 임의 창작 0. 표·수치는 원문 직접 카운트로 재확인.
- [ ] **banned token 0** — 본문 rule/criteria/notice/표에 금지 토큰·기호(`↔` 등) 없음(감사 I3와 정합).
- [ ] **to-be 무배지 0** — as-is 근거 없는 to-be 값이 `field_review` 배지 없이 본문에 박히지 않았는가.
- [ ] **카운트 불변** — `expected_counts` 일치·재빌드 byte 동일.

## 음성 테스트 (검출력 확인)
검수 에이전트/게이트가 **실제로 위반을 잡는지** 역으로 확인한다. 예: 의도적 결함(store에서 J4·K452 검출) 또는 `check_L` 비활성→활성으로 dangling 검출되는지 확인 후 원복. 검출력이 0이면 검수 자체가 무의미.

## 산출
PASS/FAIL + 발견 항목(있으면 편집 1건 루프로 수정). 검수 결과는 핸드오프 노트에 기록(예 "적대 검수 4/4 PASS").
